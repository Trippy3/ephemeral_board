# create-sticky-board

このスキルは、雑然とした情報を **Ephemeral Board が `import` で読み込める Markdown** として書き出す
ためのものです。出力は `notes` / `connectors` / `frames` の YAML フェンスブロックを並べた
1 つの `.md` ファイルです。

ユーザーは出力した `.md` を Ephemeral Board の「インポート」ボタンから読み込むだけで、
整理済みのボードが復元されます。

---

## いつ使うか

- 議事録・ブレストメモ・要件リスト・要約を **付箋として並べたい**
- 論点 / アイデア / 課題を **色やフレームで分類したい**
- 因果関係・依存関係・フローを **矢印でつなげて見える化したい**
- 「とりあえず思いついたことを箇条書きしたメモを整理して」と頼まれた

明示的に "Ephemeral Board" と言われなくても、上のような目的が読み取れたら使ってください。

逆に **使わない** ケース:

- 既存の `.md`（既にインポート済みボードのエクスポート結果）を編集したい → 普通の編集で OK
- 単にテキスト要約が欲しいだけ
- 図表（折れ線グラフなど）を描きたい → これは付箋ボードの出番ではない

## ボード設計の作法

これがこのスキルの本体です。情報を貼り付けるだけでなく、**構造化された見え方**にするためのルール。

### 1. 全体方針 — 「ぱっと見でカテゴリと関係が分かる」

- **付箋 1 枚 = アイデア 1 個 / 論点 1 個 / 要件 1 個**。文章を詰め込みすぎない（目安: 1 付箋 100 文字以内）
- 関連する付箋は **物理的に近くに置く**（後述のレイアウト規則参照）
- カテゴリは **色 + フレーム** の二段構えで示す
- 関係性（依存・原因→結果・派生）は **矢印**、対比や仲間関係は **線（style: line）** を使い分ける

### 2. 色の使い分け（カテゴリ）

利用可能な色は次の 10 色。**カテゴリと色の対応は AI が決め、ボード冒頭の "凡例フレーム" に書く**こと。

| hex | ラベル | 推奨用途 |
|---|---|---|
| `#fef08a` | Yellow | デフォルト・一般メモ |
| `#86efac` | Green | 成果 / 良かった点 / Pros / 完了 |
| `#fca5a5` | Red | 課題 / リスク / 反省点 / Cons |
| `#93c5fd` | Blue | 事実 / データ / 引用 |
| `#c4b5fd` | Purple | アイデア / 提案 / 仮説 |
| `#fdba74` | Orange | アクション / TODO / 次の一手 |
| `#f9a8d4` | Pink | 感情 / 利用者の声 |
| `#ffffff` | White | 補足・余白扱い |
| `#9ca3af` | Gray | 保留 / 参考情報 |
| `#1f2937` | Black | 見出し付箋（タイトル代わり） |

ルール:

- **同じカテゴリは同じ色**で統一（ばらつかない）
- 色を使う数は 3〜5 色程度に抑える（多すぎると意味が伝わらない）
- ユーザー入力の中に既に "Pros / Cons" や "課題 / 解決策" のようなラベルがあれば、上の表に従って色を割り当てる

### 3. フレーム（矩形）でグルーピング

- 同じカテゴリの付箋は 1 つのフレームで囲う
- フレームの `title` は **カテゴリ名そのもの**を入れる（例: "課題", "Phase 1: 設計", "ユーザーの声"）
- フレームは付箋を **完全に内包**するサイズにする（端が出ないように余白 30〜50px）
- フレーム同士は重ねない（並べる）
- フレームが不要な場合（情報が少ない、関係性主体）は無理に作らない

特殊用途のフレーム:

- **凡例 (Legend) フレーム**: ボード左上に置き、色とカテゴリの対応を見出し付箋で示す
  - 凡例には **実際にメインボードで使った "カテゴリ色" だけ** を載せる（各カテゴリ 1 枚）
  - コネクタ専用の強調色（`#ef4444` など）や、メインで使っていない色を凡例に出さない
  - 凡例タイトル付箋は `#1f2937`（黒）で OK
- **時系列フレーム**: 横並びに `Phase 1`, `Phase 2`, `Phase 3` のように複数フレームを並べる
- **マトリクスフレーム**: 2x2 などのグリッドを表現するときは 4 つのフレームをタイル状に配置

### 4. 矢印・線でつなぐ

- 付箋間に **意味のある関係** がある場合のみコネクタを使う（飾りで引かない）
- `style: arrow` — 一方向の関係（原因→結果, 親→子, 入力→出力, タスクの順序）
- `style: line` — 対比 / 双方向 / 単なる仲間関係
- `shape`:
  - `straight`: ほぼ直線。短距離 / シンプルな関係
  - `elbow`: L 字型に折れる。フローチャートや系統図向け
  - `curved`: 距離が離れている / 視覚的に他の線を避けたいとき
- 同じ付箋から多数の矢印が出ると見づらい → 出る本数は 1 付箋あたり 4 本までを目安に
- 色は基本 `#475569`（デフォルト）。**重要なクリティカルパス**だけ赤 `#ef4444` などに変えてもよい

### 5. レイアウト規則（付箋の重なりを絶対に作らない）

最も間違えやすいポイントです。座標は次の規則に従ってください。

- 1 つの付箋のサイズは **width: 200 / height: 160** をデフォルトに
- 付箋同士の最小マージン: **横 40px / 縦 40px**
  - つまりグリッドで配置する場合、x のステップは **240** (200 + 40)、y のステップは **200** (160 + 40)
- フレームは内側の付箋を全て含むように **左上 (x-30, y-50) / 右下 (max_x+30, max_y+30)** くらいの余白を取る
  - `y-50` なのはフレームのタイトルバー分
- 凡例フレームは左上 (おおむね x: 0〜400 / y: 0〜400) に置き、メインコンテンツはその右側 / 下側から開始
- 大カテゴリのフレーム同士は **80px 以上**離す

レイアウト計算は **必ず数値を二重チェック** してから書き出してください。
重なると Ephemeral Board 上で読めなくなります。

### 6. テキスト書式（HTML を埋め込める）

`text:` フィールドはプレーン文字列ですが、**実体は HTML** です。次のタグが使えます:

- `<strong>強調</strong>` / `<em>斜体</em>` / `<u>下線</u>`
- `<br>` で改行
- `<span style="color: #...">色付き</span>` / `<span style="font-weight: 600">太字</span>`
- `<ul><li>箇条書き</li></ul>`

例: `text: "<strong>要件1</strong><br>API のレートリミット強化"`

ただし:

- **`<script>` / `<iframe>` / `on*=` 属性 / `javascript:` URL は禁止** (サーバーサニタイザで除去される)
- 多用しない。1 付箋に 2-3 個までの装飾が読みやすさの限度
- 改行は `<br>` を使う（生の改行文字は YAML 的に問題が出やすいので避ける）
- YAML 内に書くため、ダブルクォートで囲み、内部のダブルクォートは `\"` でエスケープする

### 7. ID の発行ルール

- ID は **一意な英数字 8〜10 文字程度**（例: `n_meet_01`, `f_legend`, `c_a_b`）
- 接頭辞で種別が分かるようにする: 付箋 `n_`, フレーム `f_`, コネクタ `c_`
- ID は人が読んでも何の付箋か分かるように内容を反映させる（`n_login_bug` など）
- `from` / `to` で付箋 ID を参照するため、**付箋を先に定義**してから矢印を書く
- 全 ID は重複させない

### 8. createdBy / タイムスタンプ

- `createdBy` は `"AI"` 固定 (もしくはユーザーが指定した名前)
- `createdAt` / `updatedAt` は出力時の現在時刻 (ISO 8601 例: `2026-05-08T12:34:56.000Z`) で揃える
- `zIndex` は付箋の前後関係。重なりがなければ 1, 2, 3... と単調増加で OK
- フロントマターの `exported` も同じ ISO タイムスタンプで揃えると見やすい

## 進め方（実行手順）

ユーザーから情報を受け取ったら、次の流れで進めます:

### Step 1 — 情報の理解

まず情報の **構造を読み取る**:

- 何個ぐらいの主要トピックがあるか
- 自然なカテゴリ分けはあるか（Pros/Cons / 課題/解決 / Phase / People など）
- 関係性（依存 / 因果 / フロー）はあるか
- 抽出した付箋数の目安: **5〜30 枚** が読みやすい範囲。50 枚を超えそうなら「要約版を作るか / 全部出すか」をユーザーに確認

### Step 2 — レイアウト方針を決める

次のいずれか / 組み合わせで設計:

- **カテゴリ別グリッド**: フレーム = カテゴリ。各フレーム内に付箋をタイル状に並べる
- **フローチャート**: 付箋を矢印で時系列につなぐ。フレームは Phase ごと
- **マインドマップ**: 中央に主題、放射状にサブトピック
- **マトリクス**: 2x2 / 4x4 のフレームグリッド（重要度 × 緊急度 など）
- **タイムライン**: 横軸に時間、縦に並列イベント

選んだ理由をユーザーに 1-2 行で伝えてから、生成に入ると親切です。

### Step 3 — Markdown を組み立てる

1. **凡例フレームを左上に置く**（色とカテゴリの対応 + 見出し付箋）
2. **メインの大フレーム**を凡例の右に配置
3. 各大フレーム内に付箋を 240px グリッドで並べる
4. コネクタを最後に追加（付箋 ID が出揃ってから）
5. フロントマターのカウントを最終的な実数に合わせる

### Step 4 — 検証チェックリスト

書き出す前に **必ず**確認:

- [ ] 全 ID が重複していない
- [ ] 全コネクタの `from` / `to` が実在する付箋を指している
- [ ] 全フレームが配下の付箋を完全に内包する座標になっている
- [ ] 付箋同士が重なっていない（x, y がグリッド規則に沿っている）
- [ ] 色は規約に沿っている（hex 形式 `#xxxxxx`）
- [ ] **フロントマターの notes/connectors/frames 数が実際の数と一致している（後述の数え方を必ず使う）**
- [ ] YAML フェンスは ` ```yaml note ` / ` ```yaml connector ` / ` ```yaml frame ` のいずれか（タイポ厳禁）

**数え方（暗算しない）**: 頭の中の足し算は頻繁にずれます。フロントマターの数値を書く前に、
出力本文の YAML フェンスを **直接数えてください**:

- `` ```yaml note `` の出現回数 = `notes:`
- `` ```yaml connector `` の出現回数 = `connectors:`
- `` ```yaml frame `` の出現回数 = `frames:`

可能なら、書き出す前に出力文字列に対して `grep -c` 相当の機械的カウントを行い、フロントマターを
最後に書く（"テンプレ → 中身を全部書く → 中身を数える → フロントマターのカウントを埋める" の順）。
**「カテゴリごとに何枚作ったかを足し算する」アプローチは禁止** — 内訳の足し算は人間ミスを招きます。

### Step 5 — 出力

ファイル名は `board-<トピック名>-<YYYYMMDD>.md` を推奨（ユーザーが指定すればそれに従う）。
保存先はユーザー指定 / カレントディレクトリ / `out/` などの妥当な場所。

ユーザーに伝えること:

- 何枚の付箋・何本の矢印・何個のフレームを作ったか
- どんなカテゴリ分けをしたか（凡例の内容を要約）
- インポート手順: 「Ephemeral Board の右上メニューから "Import" を選んで、生成した `.md` を読み込んでください」

## よくある間違い（事前回避リスト）

過去にエージェントがこの種の出力で失敗したパターン。**書き出す前にこのリストで自己レビュー**してください。

1. **付箋が重なっている** — レイアウト計算ミス。240px グリッドを守っているか確認
2. **フレームが付箋を内包していない** — フレームの (x, y, width, height) と内部付箋の境界を再確認
3. **矢印の `from` / `to` が存在しない ID** — 付箋を消したのにコネクタを残した、ID をタイポした
4. **`color` が `red` などの英単語** — 必ず `"#xxxxxx"` の hex 表記
5. **YAML フェンスのタグ間違い** — `yaml note` であって `yaml-note` でも `note` でもない
6. **フロントマターのカウントが不一致** — 観測された失敗例: カテゴリ別に枚数を暗算で合計して
   `5 + 3 + 3 + 4 = 14` と書いたが、実際は 15。数値を書く前に必ず YAML フェンス出現数を直接数えること
7. **`text:` の中に生の改行** — YAML が壊れる。`<br>` を使うか、YAML のブロックスカラー `|` を使う
8. **`text:` の中のダブルクォートをエスケープし忘れ** — `\"` で逃がす
9. **ID が長すぎる / 空** — 1〜64 文字以内、英数字＋アンダースコアが安全
10. **フロントマターの `schemaVersion` が `1` や文字列** — 必ず `2` (整数)
