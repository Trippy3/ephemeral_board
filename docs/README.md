# Sticky Board

Miro / FigJam ライクなリアルタイム共同付箋ボードアプリケーション。
ローカルネットワーク上で動作し、HTTP トンネルを通じて社内メンバーと共同ワークできる。

## クイックスタート

### 前提条件

- Node.js v20 以上

### インストールと起動

```bash
cd sticky-board
pnpm install
pnpm dev
```

ブラウザで http://localhost:3000 を開く。

### 社内メンバーと共有する（cloudflared）

```bash
# 別ターミナルで実行
cloudflared tunnel --url http://localhost:3000
```

表示されたトンネル URL を共有すれば、社内メンバーがブラウザからアクセスできる。

### 本番風に起動する（ウォッチなし）

```bash
pnpm build   # クライアント JS をバンドル（minify）
pnpm start   # サーバーのみ起動
```

## 使い方

### ボードに参加する

1. URL を開くと名前入力ダイアログが表示される
2. 名前を入力して「Join」をクリック（Enter キーでも可）
3. ボードに参加し、他のユーザーと共同作業が開始される

### 複数ボードを使い分ける

URL パスがボード ID になる。

| URL | ボード ID |
|-----|----------|
| `http://localhost:3000/` | `default` |
| `http://localhost:3000/sprint-retro` | `sprint-retro` |
| `http://localhost:3000/brainstorm-2026` | `brainstorm-2026` |

異なる URL にアクセスすれば、独立したボードが自動作成される。

### 付箋の操作

| 操作 | 方法 |
|------|------|
| 付箋を作成 | ボードの空白部分をダブルクリック |
| テキスト編集 | 付箋内のテキスト部分をクリックして直接入力 |
| 移動 | 付箋のヘッダー（作成者名の部分）またはテキスト以外の余白をドラッグ |
| リサイズ | 付箋の右下角をドラッグ |
| 色を変更 | 付箋にホバー → 🎨 ボタンをクリック → 色を選択 |
| 削除 | 付箋にホバー → ✕ ボタンをクリック |

### 作成時の色を選ぶ

ツールバー左側のカラーパレットから色を選んでおくと、次に作成する付箋がその色になる。
7 色から選択可能: Yellow / Red / Green / Blue / Purple / Orange / Pink

### ボードの操作

| 操作 | 方法 |
|------|------|
| パン（スクロール） | ボードの空白部分をドラッグ |
| ズームイン/アウト | マウスホイール、またはツールバーの +/- ボタン |
| ズームリセット | ツールバーの `100%` ボタン |

ズームはマウスカーソルの位置を中心に拡大/縮小される（0.2x〜3.0x）。

### 他のユーザーの存在確認

- ツールバー右側にユーザーの頭文字アイコンが表示される
- ボード上に他のユーザーのカーソル（色付き矢印 + 名前ラベル）がリアルタイムで表示される

### Markdown エクスポート

1. ツールバーの「Export MD」ボタンをクリック
2. Markdown ファイルがダウンロードされる

出力形式:

```markdown
---
board: default
exported: 2026-03-25T10:30:00.000Z
notes: 8
---

## Yellow

- アイデア: オンボーディングフローの改善
- 競合の価格調査

## Red

- BLOCKER: 本番 API レート制限
- 認証リダイレクトの修正が必要

## Green

- 完了: DB マイグレーション
- 完了: CI パイプライン更新
```

- 付箋は色別にグルーピングされ、各色が見出しセクションになる
- セクション内は空間的に並び替え（上→下、左→右）
- YAML front-matter にボード ID、出力日時、付箋数を含む

## 設定・カスタマイズ

### ポート番号の変更

```bash
PORT=8080 pnpm dev
```

環境変数 `PORT` でサーバーのリッスンポートを変更できる（デフォルト: 3000）。

### 自動クリーンアップの挙動

`state.ts` 内の定数で制御:

| 定数 | デフォルト | 説明 |
|------|-----------|------|
| `CLEANUP_INTERVAL` | 10 分 | クリーンアップチェックの実行間隔 |
| `BOARD_TTL` | 24 時間 | ボードの最終操作からの保持期間 |

### 付箋の初期サイズ

`state.ts` の `createNote` 関数内:

| プロパティ | デフォルト |
|-----------|-----------|
| `width` | 200px |
| `height` | 160px |

### 付箋の色の追加/変更

以下の 3 箇所を同時に変更する:

1. `public/index.html` — ツールバーのカラーパレットボタン
2. `public/app.ts` — `showColorPicker` 関数内の `colors` 配列
3. `export.ts` — `NOTE_COLORS` のカラー名マッピング

### ユーザーカーソルの色

`state.ts` の `USER_COLORS` 配列で定義（8 色ローテーション）。

## プロジェクト構成

```
sticky-board/
  package.json          # 依存関係・pnpm scripts
  pnpm-lock.yaml        # pnpm ロックファイル
  tsconfig.json         # TypeScript 設定
  server.ts             # Express + Socket.IO サーバー、API routes
  state.ts              # インメモリ状態管理（CRUD, TTL cleanup）
  export.ts             # Markdown エクスポートロジック
  docs/
    README.md           # このドキュメント
  public/
    index.html          # 単一 HTML ページ（ツールバー、ボード、名前ダイアログ）
    app.ts              # クライアント TypeScript ソース
    app.js              # esbuild によるバンドル済み JS
    app.js.map          # ソースマップ
    style.css           # 全スタイル定義
```

### pnpm scripts

| コマンド | 動作 |
|---------|------|
| `pnpm dev` | サーバー（tsx watch）+ クライアントビルド（esbuild watch）を同時起動 |
| `pnpm build` | クライアント JS を minify ビルド |
| `pnpm start` | サーバーのみ起動（ウォッチなし） |

## アーキテクチャ概要

```
┌─────────────────────────────────────────────────────┐
│                    Browser (Client)                  │
│                                                     │
│  index.html + app.js + style.css                    │
│  ├── DOM 操作で付箋を描画（div + contenteditable）  │
│  ├── Pointer Events API でドラッグ/リサイズ         │
│  ├── Socket.IO Client で双方向通信                  │
│  └── オプティミスティック更新（操作を即座に反映）   │
└────────────────────┬────────────────────────────────┘
                     │ WebSocket (Socket.IO)
                     │
┌────────────────────┴────────────────────────────────┐
│               Node.js Server (server.ts)            │
│                                                     │
│  Express                                            │
│  ├── 静的ファイル配信（public/）                    │
│  └── GET /api/boards/:boardId/export.md             │
│                                                     │
│  Socket.IO                                          │
│  ├── board:join → 全状態同期（board:sync）          │
│  ├── note:* → CRUD → ブロードキャスト               │
│  ├── cursor:move → 他ユーザーへ転送                 │
│  └── disconnect → ユーザー退出通知                  │
│                                                     │
│  state.ts（インメモリ Map）                         │
│  ├── boards: Map<boardId, BoardState>               │
│  └── 10 分ごとに TTL チェック、24h 経過で自動削除   │
└─────────────────────────────────────────────────────┘
```

### リアルタイム同期の流れ

1. ユーザー A が付箋を操作 → クライアントがオプティミスティックに UI を即時更新
2. Socket.IO でサーバーにイベント送信（例: `note:move`）
3. サーバーがインメモリ状態を更新
4. サーバーが他の全クライアントにブロードキャスト（例: `note:moved`）
5. ユーザー B のクライアントが受信して DOM を更新

### テキスト編集の同期

- クライアント側で 300ms デバウンス後にサーバーへ送信
- 受信側は、そのテキストエリアにフォーカスしていなければ内容を上書き
- フォーカス中（自分が編集中）なら上書きしない → 自分の入力が消えない

## 技術的意思決定の背景

### なぜ Vanilla TypeScript か（React / Svelte を使わない理由）

付箋ボードは本質的に単一画面のアプリケーションで、ルーティングや複雑なコンポーネントツリーが不要。付箋は `position: absolute` の `div` 要素で十分に表現でき、テキスト編集は `contenteditable` 属性でブラウザのネイティブ機能をそのまま利用できる。

フレームワークを導入すると:
- ビルドパイプライン（Vite 等）のセットアップが必要になる
- ドラッグ操作で React の再レンダリングとの戦いが発生しうる
- バンドルサイズが増加する

Vanilla TS なら esbuild で単純にバンドルするだけで済み、DOM 操作も直接的で予測しやすい。

### なぜ Socket.IO か（Native WebSocket / SSE を使わない理由）

Native WebSocket は軽量だが、以下の機能を自前で実装する必要がある:
- 自動再接続（指数バックオフ）
- ルーム管理（ボード別のブロードキャスト）
- トランスポートフォールバック（WebSocket が使えない環境でのポーリング）

Socket.IO はこれらを組み込みで提供しており、クライアントライブラリも対応するため開発コストが大幅に削減される。
SSE は単方向（サーバー→クライアント）であり、クライアントからの送信には別途 HTTP POST が必要になるため、リアルタイム性が落ちる。

### なぜ LWW（Last Write Wins）か（CRDT / ロック方式を使わない理由）

付箋のテキストは一般的に 1〜3 文程度の短い内容であり、同一の付箋を 2 人が同時に編集するケースは稀。
- **CRDT（Yjs 等）**: 文字レベルのマージが可能だが、ライブラリの導入と状態管理の複雑さがこの規模には不釣り合い
- **ロック方式**: 確実だが「誰かが編集中で触れない」という UX が硬い

LWW なら実装がシンプルで、付箋という用途には実用上問題ない。テキスト編集の 300ms デバウンスにより、細かい打鍵ごとの競合も抑制される。

### なぜ DB を使わないか（インメモリのみ）

このツールは「一時的なワークスペース」として設計されている:
- ワークショップやレトロスペクティブなど、セッション単位で使う想定
- 必要な成果物は Markdown エクスポートで保存すれば十分
- DB を入れるとセットアップ手順が増え、「`pnpm dev` で即起動」の手軽さが失われる
- サーバー再起動や 24 時間経過でデータが消えるのは意図的な仕様

### なぜモノリス構成か（フロントエンド/バックエンド分離しない理由）

- 起動コマンドが 1 つで済む（`pnpm dev`）
- CORS 設定が不要（Express が静的ファイルも配信）
- HTTP トンネルで公開するポートが 1 つで済む
- この規模のアプリケーションで分離するメリットがない

### なぜ esbuild か（webpack / Vite を使わない理由）

- クライアント側は単一ファイル（`app.ts`）→ 単一バンドル（`app.js`）のシンプルな構成
- esbuild はゼロコンフィグでバンドル可能（CLIオプションだけで完結）
- ビルド速度が極めて高速（41ms）
- Vite は開発サーバーを立てる前提があり、Express との共存に追加設定が必要

### なぜ cloudflared か（ngrok を使わない理由）

- 無料で安定して利用可能
- Cloudflare Access と連携すればトンネルに認証をかけられる
- 社内ネットワークでの利用に適している

ただし、アプリケーション自体はトンネルツールに依存しないため、ngrok やその他のツールでも動作する。

## 制限事項と今後の拡張

### 現在の制限事項

- **永続化なし**: サーバー再起動で全データが消える（仕様）
- **認証なし**: URL を知っていれば誰でもアクセスできる（トンネル側の認証で制御する想定）
- **テキスト競合**: 同一付箋の同時編集は LWW（後から書いた方が勝つ）
- **最大規模**: DOM ベースのため、1 ボード数百枚程度が実用上限

### 拡張の候補

| 拡張 | 実装の方針 |
|------|-----------|
| 画像の貼り付け | `note:create` にファイルアップロードを追加、Base64 でインメモリ保持 |
| 付箋のグルーピング | 枠線 `div` を追加、内部の付箋をグループ単位で移動 |
| テンプレートボード | 初期状態の付箋配置を JSON で定義、`/template/retro` 等で起動 |
| 投票機能 | 付箋に投票カウンターを追加、Socket.IO イベントで同期 |
| JSON エクスポート/インポート | 全ボード状態を JSON で保存・復元（セッション間のデータ引き継ぎ） |
