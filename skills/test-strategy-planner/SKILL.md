# Test Strategy Planner

このスキルは「**テスト戦略を立てて提示する**」工程を担当する。実装は別タスク扱い。

## なぜこのスキルがあるのか

普通の「テストを書いて」依頼に素直に従うと、ROI を考えずに片っ端からテストを書いてしまい、
- 表面的なカバレッジ率は上がるが回帰検出力は弱い
- 内部実装に紐づいた brittle なテストが量産される
- 一番痛い壊れ方 (round-trip 不整合・XSS・LWW 不変条件など) は素通りする

という典型的な失敗パターンに陥る。このスキルは、参考記事から抽出した共通解
(coverage 率は KPI にしない / Integration を最厚に / テストサイズで考える) を踏まえ、
プロジェクト固有の「壊れたら一番痛い動作」を起点に 3〜4 層を設計する。

## いつ実装まで踏み込むか

このスキルは **計画と提示までで止める**。
PR 分割・実装は人間 (or 別スキル) に委ねる。理由:
- 計画の質と実装の正確性は別の能力で、ひとつのスキルに混ぜると指示が太くなる
- 計画書だけならドメイン汎用、実装は言語/ランナー依存
- 計画段階で 1 度人間の合意を取ると、その後の実装ブレが激減する

ユーザーが「実装まで一気に」と明示した場合のみ、計画提示後に確認をとってから実装に入る。

## 想定するワークフロー (5 フェーズ)

各フェーズは**並列化できるところは並列化する**。Explore subagent や Plan subagent が
使えるなら積極的に使う。1 フェーズ 1 ステップではなく、独立した調査は 1 メッセージ内で
複数ツール呼び出しを並べる。

### Phase 1. 参考記事を読む (任意・与えられたとき)

ユーザーがテスト戦略系の URL を提示したら、`WebFetch` (subagent でも本体でも) で
**並列**に読み、共通解と固有の主張を 1000 字程度で構造化要約する。

抽出する観点 (テンプレ):
- 「カバレッジ率 vs 信頼性 vs 保守性」のうち何を優先すべきと言っているか
- 各層 (unit/integration/e2e) の比率や役割
- 壊れにくく書くコツとアンチパターン (snapshot / 内部実装依存 / 自作自演)
- t_wada さんなど、テストサイズ (small/medium/large) の話があれば必ず拾う

参考記事が無いとき、`references/principles.md` に内蔵した共通解を土台に進める。

### Phase 2. コードベースをインベントリする

`Explore` subagent (or 並列の `grep` + `Read`) で、テスト対象になり得るコードを
**5 つの分類**で棚卸しする:

1. **純粋ロジック** (副作用なし・入出力決定論的) — 最もテストしやすい
2. **状態 + 副作用ありロジック** (Map/Set 内部状態に依存・DI 可能) — 中粒度
3. **IO バウンダリ** (HTTP / WebSocket / DOM / 子プロセス) — Integration / E2E
4. **「壊れたら一番痛い」コア機能** — このスキルの心臓部 (次節)
5. **テストしづらい領域** — グローバル DOM / 外部バイナリ / リアルタイム性。ROI 低なら割り切る

各関数は `path/to/file.ts:42` 形式で行番号付き引用する。あとで計画書から逆引きできるように。

### Phase 3. 「壊れたら一番痛い」を 5〜8 個に絞る

このスキルの最も重要な工程。`docs/`・`README.md`・`CONTRIBUTING.md`・`AGENTS.md`・
git log・既存の落とし穴注意書きを読み、**プロジェクト固有の不変条件**を抽出する。

良い「壊れたら痛い」項目の特徴:
- 一行で説明できる契約 (例: 「付箋削除時はそれに繋がるコネクタも同時に削除される」)
- 失敗した時にユーザーが即座に気づく or データを失う
- 内部リファクタで壊しがち (LWW、cascade、ID 保持、サニタイズの 2 段防御 など)

良くない項目の例:
- 「ボタンの色が正しい」「タイトルが表示される」(snapshot で済む / 戦略レイヤーじゃない)
- 「全機能が動く」(粒度が粗すぎて担保不可能)

最後に **traceability 表** を作る。これが計画書の核:

```markdown
| 不変条件 | 担保するテスト |
|---------|---------------|
| 1. ... | L3 xxx.test.ts + L4 #2 |
```

### Phase 4. 4 層を設計する

```
┌─────────────────────────────────────────────────┐
│ L4. E2E              ─ 3〜5 シナリオ    Large    │ ← 最小限
├─────────────────────────────────────────────────┤
│ L3. Integration      ─ HTTP/Socket    Medium    │ ← 厚め
├─────────────────────────────────────────────────┤
│ L2. Unit (Pure)      ─ 決定論的ロジック         │ ← 厚め
├─────────────────────────────────────────────────┤
│ L1. Static           ─ 型 / Linter              │ ← 既存
└─────────────────────────────────────────────────┘
```

各層について書くこと:

- **狙い**: 何を捕まえる層か (1 行)
- **対象**: 具体的なファイル名・関数名と「テストの why」(snapshot ではなく属性別 assertion で…等)
- **書き方ルール**: 内部実装依存の禁止・モック範囲・テストサイズ感
- **コスト感**: 何 ms / 何 sec で完了する想定か

「対象」表は **1 行 1 ファイル**。1 関数ごとではなく機能単位でまとめる。

### Phase 5. ロールアウト計画を作る

テスト戦略を 1 PR で実装するのは現実的でない。**3〜5 PR に分割**し、各 PR が単独で
CI を通る形に分ける。例:

1. **PR-1: 足回り** — テストランナー導入 / 設定ファイル / smoke test
2. **PR-2: L2 Unit** — 純粋ロジック層 (一気にコア機能の 4〜5 割をカバー)
3. **PR-3: L3 Integration** — 残るコア機能を網羅
4. **PR-4: L4 E2E** — ブラウザ実機でしか取れないバグ用 (最小限)

各 PR の **完了条件** (CI 通過・テスト件数・対象不変条件) を明記する。

## 出力テンプレート

最終提示は `references/plan-template.md` のセクション順を厳守する。
セクションを増やしても良いが、**減らしてはいけない**。減らすとレビュアが
「重要な観点が漏れていないか」を判定できない。

ユーザーが「実装まで」と言うまでは **plan モード** に入って plan ファイルに
書き出すのが望ましい (Claude Code 環境であれば)。

## アンチパターン (このスキルが防ぎたいこと)

- **カバレッジ率を KPI にする**。代わりに「壊れたら一番痛い」項目に対応するテストの
  存在を KPI にする
- **DOM の snapshot test**: CSS / フォント / Pointer API に依存しすぎてフレーク源になる
- **socket / DB / queue 内部のモック**: 「自作自演テスト」(t_wada) になる。本物の
  in-memory transport / test container を使う
- **1 関数 = 1 テストファイル**: 機能単位で 1 ファイルにする。"export 周り全部"
  のように
- **brittle なセレクタ** (nth-child / class-name 細部 / xpath): freee の事例が示す通り
  すぐ壊れる。`getByRole`/`getByText` 系で意味で指す
- **計画段階で実装 PR を 1 つに混ぜる**: PR が太ると判断ミスが連鎖する
- **「将来のために」抽象化**: テスト戦略はコードベースの今に密着しているべき。
  半年後の機能のために柔軟な設計を入れない

## 進め方の指針 (Claude 向け)

- **計画は 1 度で確定しようとしない**。Phase 3 後に「これで合っていますか？」と
  AskUserQuestion を挟むと精度が大きく上がる (特に「壊れたら一番痛い」項目はドメイン
  知識を含むので人間との合意が肝)
- **ツール選定は判断ポイントだから AskUserQuestion で**。ランナー (Vitest / Jest /
  node:test)、E2E (Playwright / Cypress)、CI ゲート粒度などはトレードオフがある
- **「読む」と「考える」を分ける**。インベントリ後にすぐ計画を書き始めない。一度頭の
  中で 4 層を組み立てて、それから書く
- **最初に出すテンプレは長くて良い**。短くしすぎると判断材料が足りなくなる。プラン
  確定後にユーザーに不要部分を削ってもらえる
